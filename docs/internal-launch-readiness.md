## 产品验收结论

核心功能范围：

- 销售生成器：客户信息、行业标签、产品匹配、折扣/一口价、申请理由、复制邮件。
- 产品报价规则后台：登录、产品类型、报价规则、行业匹配、变更记录。

阻断性问题：无新增业务功能阻断；上线阻断集中在部署安全和访问边界。

### 内部用户使用手册（极简版）

销售生成邮件：

1. 打开内部试用地址。
2. 填客户公司、客户腾讯云账号 ID、商机编号、申请日期。
3. 选择行业标签，系统生成推荐产品。
4. 产品不全时点“新增产品”，输入产品关键词后手动补齐。
5. 确认计费方式、申请方式、政策档位、折扣/单价、返佣。
6. 填成立年份、公司规模、目前项目情况，点“生成理由”。
7. 看右侧阈值提醒，重点确认重复产品、未匹配产品、低于突破政策。
8. 点“复制标题”“复制正文”，粘贴到邮件。

商务维护规则：

1. 打开 `/admin`，用个人账号登录。
2. 搜索产品，编辑折扣、返佣、一口价、计费方式和说明。
3. 缺产品时新增产品，并补别名。
4. 在行业匹配页维护行业默认推荐产品。
5. 改完后让销售刷新生成器页面验证。

常见问题：

| 问题 | 处理 |
| --- | --- |
| 复制正文提示缺字段 | 补客户账号、产品名称、折扣/单价、计价单位或返佣 |
| 提示商务库未登记 | 先手动填，反馈商务补规则 |
| 推荐产品太多 | 删除不申请的产品，只保留本次申请项 |
| 折扣/返佣不确定 | 先找商务确认，不要直接发 |

## 后端部署说明

环境要求：Ubuntu 22.04 或同级 Linux、Python 3.10+、Nginx、systemd。

关键环境变量：

```bash
POSTEMAIL_DB=/opt/postemailagent/data/rules.db
POSTEMAIL_SECRET=<强随机值>
POSTEMAIL_ADMIN_PASSWORD=<首次初始化管理员密码，可选>
```

初始化：

```bash
cd /opt/postemailagent
python3 -m venv venv
./venv/bin/pip install -r server/requirements.txt

POSTEMAIL_DB=/opt/postemailagent/data/rules.db \
POSTEMAIL_ADMIN_PASSWORD='<强密码>' \
./venv/bin/python -m server.scripts.import_products
```

启动验证：

```bash
sudo systemctl daemon-reload
sudo systemctl enable postemailagent
sudo systemctl restart postemailagent
curl -f http://127.0.0.1:5050/api/health
```

备份：

```bash
POSTEMAIL_DB=/opt/postemailagent/data/rules.db \
POSTEMAIL_BACKUP_DIR=/opt/postemailagent/backups \
/opt/postemailagent/venv/bin/python -m server.scripts.backup_db
```

建议每天 02:00 执行一次，保留 14 天，并每周异地保存一次。

## 测试与验收

### 核心回归

- `/` 能加载行业、产品和行业产品映射。
- 邮件标题和正文能随输入更新。
- 行业标签多选后自动带出产品，重复产品去重。
- 未匹配、低于突破政策、重复产品能提示。
- `/admin` 未登录跳 `/login`。
- `admin` 可操作全部后台能力，`business` 不可操作管理员能力。

### 50 并发小压测建议

| 场景 | 占比 | 目标 |
| --- | ---: | --- |
| 首页和静态资源 | 30% | 无 5xx |
| 基础数据读取 | 45% | GET p95 < 500ms |
| 产品匹配 | 20% | POST `/api/match` p95 < 800ms |
| 登录和后台读取 | 5% | 错误率 < 1% |

压测 5-10 分钟，1 分钟爬坡。后台写接口只压 staging DB，不压正式库。

## 回滚方案

触发条件：

- 登录失败、后台无法保存、核心生成流程不可用。
- 连续 5xx、服务频繁重启、数据库锁死。
- 生成邮件出现明显业务错误且影响审批。
- 发现公网暴露或账号风险。

回滚步骤：

1. 运维关闭入口：移除试用 IP 或临时下线 Nginx 站点。
2. 停服务：`systemctl stop postemailagent`。
3. 恢复代码目录到上线前版本。
4. 解压并恢复 `/opt/postemailagent/data/rules.db`。
5. 重启服务和 Nginx。
6. 验证 `/api/health`、登录、首页生成邮件。
7. 通知试用群系统已回退，问题冻结。

目标：RTO 30 分钟内。
